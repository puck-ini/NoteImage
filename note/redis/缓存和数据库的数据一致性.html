# 如何保持缓存和数据库的数据一致性

# 等待过期时间

直接等缓存中的 key 过期，之后请求访问这个 key 时会从数据库中取出最新的数据更新到缓存中。

优点：

  • 实现简单，出问题概率较小
  • 直接使用缓存中的过期机制，易于实现

缺点：

  • 完全依赖过期时间，如果读请求频繁，但是过期时间设置较长会读到很多脏数据。
  • 过期时间不好把控，太对容易造成缓存频繁失效，太长容易较长时间数据不一致

# 数据更新后删除缓存中旧数据

数据库中的数据更新后直接删除缓存中的数据，这样下次请求过来是缓存中数据不存在就会访问数据库中的新数据，同时将新数据保存在缓存中。主动删除缓存中的数据是为了减少不一致的时间。

优点：

  • 相对等待过期时间到达，直接删除缓存可以让数据不一致的时间更短
  • 实现成本低

缺点：

  • 如果数据库中更新成功，缓存中删除失败就会变成只能等待缓存的过期时间到达

# 通过消息队列更新缓存

数据库更新数据之后发送一个消息，让缓存监听该消息，在收到消息之后更新缓存。

优点：

  • 使用消息队列可以进行解耦，数据库更新数据成功同时成功发送消息之后就不需要关心后续的结果，消息队列的可靠性保证了缓存中的数据一定会更新

缺点：

  • 引入消息队列增加了成本

# 先更新缓存再更新数据库

在需要更新数据时先更新缓存再更新数据库。

优点：

  • 实现简单
  • 即使更新缓存成功，更新数据库失败，后续请求访问也是会获取到缓存中的新数据，短期内不会对业务造成太大影响。

缺点：

  • 如果更新缓存成功，更新数据库失败，在缓存失效时会从数据库读取旧值保存在缓存中，这就会对业务造成影响

# 解析 binlog

如果使用的时 MySQL，可以搭建一个服务通过解析 MySQL 中 binlog 日志然后将数据更新到缓存。

优点：

  • 更新数据库之后不需要做额外的操作

缺点：

  • 需要额外搭建一个服务，增加了成本
  • 同步服务如果压力较大崩了会造成较长时间的数据不一致
